1. Protocol Overview

SACP (Structured AI Collaboration Protocol) is an open, extensible, standardized AI-native technical protocol that defines structured data models, interaction specifications, and security guidelines for AI-driven cross-system collaboration. Any software (traditional office software, AI tools, business systems, developer tools) that follows this protocol can achieve seamless AI collaboration, data interoperability, and unified result accumulation.

1.1 Protocol Objectives

  • Solve the industry pain points of disordered AI output, inconsistent formats, and inability to interoperate across systems
  • Provide a ready-to-use standardized AI collaboration solution for all software, reducing AI collaboration development and integration costs
  • Build an open AI collaboration ecosystem to achieve "standardize once, collaborate everywhere"

1.2 Scope of Application

  • Traditional software: OA systems, financial reimbursement, travel management, ERP, CRM, knowledge bases
  • AI-native tools: LLM applications, AI execution tools, AI knowledge bases, BI data analysis tools
  • Developer tools: code hosting, CI/CD, project management tools
  • Personal productivity tools: todo lists, calendars, note-taking software

2. Core Design Principles

  1. Strong Core Constraints, Weak Extension Invasion: Core data models are strictly standardized to ensure interoperability; extension fields are completely open to adapt personalized needs without breaking core structures.
  2. Security Native: The protocol natively defines data security, permission isolation, and encrypted transmission guidelines to fundamentally avoid AI collaboration security risks.
  3. 100% Backward Compatible: All protocol iterations maintain complete backward compatibility of core data models; existing integrators require no code changes to adapt to new versions.
  4. Minimal Integration: The protocol provides standardized API interfaces and multi-language SDKs. Integrators only need to complete data format conversion to achieve protocol adaptation with minimum development cost.

3. Common Field Specifications

All 7 data structures include the following common fields:

Field Type Required Description
type string Yes Data type: task/report/output/message/approval/schedule/metric
title string Yes Title, max 100 characters
create_time string Yes Creation time in ISO 8601 format: yyyy-MM-dd HH:mm:ss
update_time string Yes Update time in ISO 8601 format
creator string No Creator ID/identifier
source string No Data source system identifier (e.g., OpenClaw, Enterprise OA)
extend object No Standardized extension field for personalized needs without breaking core structure

4. Seven Core Data Structures

4.1 task - Task Execution

Core Positioning: Represents tasks to be executed, todo items, development tasks, etc.

Field Type Required Description
description string Yes Detailed task description
priority string Yes Priority: 高/中/低 (High/Medium/Low)
status string Yes Status: todo/doing/confirm/done/archived
progress integer Yes Progress percentage, 0-100
from_command string No Original command that generated this task
output_expect string No Expected output or deliverable description

4.2 report - Document Report

Core Positioning: Represents analysis reports, proposal documents, summaries, knowledge accumulation, etc.

Field Type Required Description
tags array[string] No List of tags
outline array[string] No Outline (chapter title list)
summary string No Summary/Introduction
content string Yes Complete content in Markdown format
from_task string No Associated task ID (if generated from a task)

4.3 output - Output Delivery

Core Positioning: Represents deliverables such as code, files, images, links, etc.

Field Type Required Description
category string Yes Category: code/file/image/link
content string Conditional When category is code or link, stores code text or URL; otherwise can be empty
language string No Code language (recommended when category=code)
url string Conditional When category is file/image, file access URL; when category=link, link address
file_size integer No File size (bytes), only valid for file/image
from_task string No Associated task ID

4.4 message - Notification Message

Core Positioning: Used for short notifications, status reminders, etc., not carrying core business data.

Field Type Required Description
text string Yes Notification content, max 2 lines (recommended within 140 characters)

4.5 approval - Approval Process

Core Positioning: Represents various approval applications (reimbursement, leave, contract, etc.).

Field Type Required Description
apply_type string Yes Application type, e.g., 报销/请假/合同
applicant string Yes Applicant ID
approver_list array[string] Yes Approver ID list (in order)
approval_data object Yes Approval business data (specific content varies by apply_type)
current_step integer Yes Current approval step (starting from 0)
status string Yes Status: pending/approved/rejected

4.6 schedule - Schedule Management

Core Positioning: Represents meeting arrangements, schedule plans, time reminders, etc.

Field Type Required Description
start_time string Yes Start time in ISO 8601 format
end_time string Yes End time in ISO 8601 format
location string No Location (physical address or online link)
participants array[string] Yes Participant ID list
reminder_type string No Reminder type: none/email/sms/push
reminder_time string No Reminder advance time (e.g., 15min, 1hour)

4.7 metric - Metric Data

Core Positioning: Represents quantitative metrics, statistical data, performance monitoring, etc.

Field Type Required Description
metric_name string Yes Metric name
metric_value number Yes Metric value
unit string No Unit (e.g., %, ms, count)
timestamp string Yes Data timestamp in ISO 8601 format
dimensions object No Dimension information (e.g., {"region": "CN", "env": "prod"})

5. Interaction Modes

SACP supports two interaction modes:

5.1 RESTful API (Active Invocation)

System A actively calls System B's API to push SACP data:

POST https://system-b.com/api/sacp/receive

5.2 Webhook (Event Push)

System B registers a webhook URL with System A. When System A generates SACP data, it actively pushes to System B:

POST https://system-b.com/webhook/sacp

6. Security Specifications

  1. Signature Verification: All SACP data transfers must include signature verification to prevent data tampering.
  2. HTTPS Mandatory: All API calls must use HTTPS protocol to ensure transmission security.
  3. Data Isolation: Different tenants/organizations' data must be strictly isolated.
  4. Permission Minimization: Follow the principle of least privilege for API access control.
  5. Rate Limiting: Implement reasonable rate limiting to prevent abuse.
